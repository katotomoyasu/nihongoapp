# 日本語教師試験対策問題演習サイト 仕様書

version 0.2 / 2026-07-11 更新（出題数の50問単位化、ブラウザ版アップロード・変換ツールを追加）

## 1. 概要

### 1.1 目的
登録日本語教員試験（日本語教師試験）の受験対策として、4択形式の練習問題をWebブラウザ上で繰り返し解けるサイトを構築する。問題データはExcelでローカル作成し、変換スクリプトで問題データファイルへ変換したうえでサイトに反映する。バックエンドのデータベースは使用せず、静的サイト＋ブラウザ内ストレージで完結させる。

### 1.2 想定利用者
- 本人（1〜少数名）が自分の学習用に利用する私的な学習サイト
- ログイン機能・アカウント管理は行わない（同一ブラウザ＝同一利用者という前提）

### 1.3 非機能要件のポイント
- データベース不使用。問題データは静的JSONファイル、成績データはブラウザの `localStorage`（将来的にデータ量が増えた場合は `IndexedDB` へ移行）に保存する。
- さくらVPS上のnginxで静的ファイルとして配信する（サーバーサイドの動的処理は行わない）。
- 問題作成はローカルのExcelで行う。変換は「Node CLI（Claude Code経由）」と「サイト上のブラウザ変換ツール」の2通りを用意するが、いずれもサーバーサイド処理は伴わない（5.6節参照）。最終的にビルド・デプロイして本番サイトへ反映する手順は変わらない。

## 2. データ設計（DB代替方式）

### 2.1 問題データ（静的・読み取り専用）

問題データは `questions.json` としてサイトに同梱する静的ファイル。ExcelからNode.jsスクリプトで変換して生成する（3章参照）。

```jsonc
{
  "version": "2026-07-11",
  "questions": [
    {
      "id": "s3-gengo-shakai-001",     // 一意ID。{科目slug}-{中分類slug}-{元Excelの No 3桁} で生成（5.2節参照）
      "no": 1,                         // 元ExcelでのNo（同一ファイル更新時の突合キー）
      "category": "社会・文化・地域",     // 大分類（元ファイル名・管理データシートの「タイトル」から抽出）
      "subcategory": "言語と社会",       // 中分類（同上）
      "difficulty": 1,                 // 元Excelの「難易度」列（★=1, ★★=2, ★★★=3）
      "text": "「難民の地位に関する条約」における難民の定義に含まれない理由はどれか。",
      "choices": ["人種", "宗教", "経済的困窮", "政治的意見"],
      "correctIndex": 2,               // 0始まり。元Excelの「管理データ」シートの正解列から取得（後述）
      "explanation": "難民条約（1951年採択）では……",  // 元Excelの「解説（選択後に表示）」列
      "source": "教程1"                // 元Excelの「出典」列（任意項目）
    }
  ]
}
```

設計上の注意点：
- `id` は成績データ（localStorage側）と紐付けるキーになるため、**一度公開したら原則変更しない**。同一カテゴリのExcelを差し替える場合も `{科目slug}-{中分類slug}-{No}` の組み合わせが変わらない限りIDは維持される（IDを変えると過去の正誤履歴が引き継がれず「未回答」扱いに戻ってしまうため、ファイル名の日付部分はID生成に使わない設計にしている）。
- `category`／`subcategory` はExcelの各行には列として存在せず、ファイル単位（1ファイル＝1カテゴリ）でファイル名または「管理データ」シートの「タイトル」から抽出する（5.2節参照）。
- 問題数が増えてきた場合は `questions.json` を `questions/{category}.json` のようにカテゴリ別分割し、初期表示の読み込みを軽くすることも検討（初期は単一ファイルで問題ない）。

### 2.2 成績・学習履歴データ（ブラウザ側）

`localStorage` に以下のキーで保存する。

```jsonc
// key: "jlt_progress_v1"
{
  "schemaVersion": 1,
  "records": {
    "s3-gengo-shakai-001": {
      "attempts": 3,            // 累計回答回数
      "correct": 2,             // 累計正解回数
      "lastResult": "correct",  // 直近の正誤 "correct" | "incorrect"
      "lastAnsweredAt": "2026-07-10T09:12:00+09:00",
      "history": [               // 直近N件のみ保持（例:直近10件）でデータ肥大化を防ぐ
        { "at": "2026-07-10T09:12:00+09:00", "result": "correct" }
      ]
    }
  },
  "sessions": [                  // 演習セッション単位のログ（グラフ表示用）
    {
      "id": "S0001",
      "mode": "review_wrong",     // "all" | "category" | "review_wrong" | "mock_exam"
      "startedAt": "2026-07-10T09:00:00+09:00",
      "finishedAt": "2026-07-10T09:12:00+09:00",
      "questionIds": ["s3-gengo-shakai-001", "s3-gengo-shakai-002"],
      "correctCount": 8,
      "totalCount": 10
    }
  ]
}
```

設計上の注意点：
- `schemaVersion` を持たせ、将来データ構造を変える際にマイグレーション処理（旧形式→新形式変換）をアプリ起動時に実行できるようにする。
- `localStorage` の容量上限（ブラウザにより約5MB程度）を踏まえ、`history` は問題ごとに直近数件のみ保持し、詳細な全履歴が必要なら `sessions` 側に持たせる設計とする。
- **バックアップ／端末間移行機能**：DBがない構成では端末やブラウザを変えると成績が消えるため、設定画面に「成績データをJSONファイルとしてエクスポート／インポート」機能を用意する（必須級の機能として3.6節に記載）。

## 3. 機能要件

### 3.1 出題・演習機能
- 4択形式で出題し、選択後すぐに正誤と解説を表示する（即時フィードバック）。
- 出題順はシャッフル（ランダム）をデフォルトとし、設定で「順番通り」にも切り替え可能とする。
- 1問ごとに「次の問題へ」で進む形式（模擬試験モードのみ後述の通り一括採点も検討）。

### 3.2 出題モード
| モード | 内容 |
|---|---|
| 全範囲演習 | 全問題からランダム出題 |
| カテゴリ別演習 | 大分類（・中分類）を選択して、その範囲のみ出題 |
| 苦手克服モード | 「直近の回答が不正解」または「正答率が閾値（例：60%）未満」の問題のみを抽出して出題。優先度は正答率が低い順・不正解回数が多い順で並べる |
| 模擬試験モード（将来拡張・任意） | 出題数と制限時間を指定し、本番形式に近い形で一括採点する |

**1セッションあたりの出題数は50問固定とする**（3.5節）。科目・中分類ごとの問題数が100問・200問規模になってきているため、1回の演習が長くなりすぎないようにする。

- 全範囲演習・カテゴリ別演習：出題プールをシャッフルし、先頭50問を出題（プールが50問以下ならプール全体を出題）
- 苦手克服モード：下記の抽出ロジックで優先度順に並べたリストの**上位50問**を出題（プールが50問以下ならプール全体を出題）

苦手克服モードの抽出ロジック（初期実装案）：
1. `records` の中から `attempts >= 1` かつ (`lastResult === "incorrect"` または `correct/attempts < 0.6`) を抽出
2. 未出題の問題は対象外（そもそも間違えた実績がないため）
3. 正答率が低い順に並べ、上位50問を出題対象とする
4. 抽出結果が0件の場合は「苦手な問題はありません」と表示し、全範囲演習を提案

### 3.3 成績・進捗機能
- 全体正答率（累計）
- カテゴリ別正答率（レーダーチャートまたは横棒グラフ想定）
- 直近の演習セッション履歴（日時・モード・正答数/問題数）
- 苦手問題ランキング（不正解回数が多い順のTop N）
- 学習日数・総回答問題数などのサマリー表示

### 3.4 問題表示・UI要件
- レスポンシブ対応（スマートフォン・PC両対応。すきま時間の学習を想定）
- 選択肢はタップ/クリックしやすい大きさを確保
- 正誤時に色分け（正解=緑、不正解=赤、選んだ誤答と正答の両方をハイライト）
- 解説文はMarkdown簡易記法（改行・強調程度）に対応できると問題作成側が書きやすい（任意）

### 3.5 設定・その他機能
- 出題数は3.2節の通り50問固定（ユーザーが自由指定できる設定は初期リリースでは持たない。将来要望があれば設定項目化を検討）
- 出題範囲の複数カテゴリ選択
- ダークモード（任意・優先度低）

### 3.6 データバックアップ機能（必須）
- 「成績データのエクスポート」：`localStorage` の内容をJSONファイルとしてダウンロード
- 「成績データのインポート」：エクスポートしたJSONファイルを読み込み、`localStorage` に復元（端末変更・ブラウザキャッシュ削除時の保険、複数端末を使う場合の擬似同期にも利用可能）
- 「成績データのリセット」：確認ダイアログ付きで初期化

## 4. 画面構成（サイトマップ）

```
/                     … トップ（モード選択：全範囲／カテゴリ別／苦手克服／設定）
/quiz?mode=...        … 出題画面（1問ずつ回答→即時採点）
/result               … セッション終了後の結果画面（正答数、間違えた問題一覧へのリンク）
/dashboard            … 成績ダッシュボード（全体・カテゴリ別正答率、苦手ランキング）
/import               … 問題データの取り込み（Excelアップロード→ブラウザ内変換→questions.jsonダウンロード、5.6節）
/settings             … 出題設定、データのエクスポート/インポート/リセット
```

SPA（1ページ内での画面遷移）として実装し、ルーティングはハッシュベース（`#/quiz` 等）とすることで、nginx側の静的配信設定をシンプルに保つ（後述6章）。

## 5. 問題データ作成・変換パイプライン（Excel → JSON）

### 5.1 運用フロー
```
① ローカルExcelで問題作成（既存の問題作成スキルを利用）
        ↓
② 変換スクリプト（Node.js）を実行し、Excel → questions.json に変換
        ↓
③ 変換スクリプトがバリデーションを実施（後述5.3）。エラーがあれば①へ差し戻し
        ↓
④ 生成された questions.json をフロントエンドのビルドに含めてビルド
        ↓
⑤ ビルド成果物をさくらVPSへ転送（rsync/scp）し、nginxが配信
```
「ローカルで作ってアップロード」の基本フローは①〜⑤の手動デプロイ運用を指す。これに加えて、Node CLIやClaude Codeを介さずその場でExcelを変換したい場合のために、サイト上にブラウザ完結の変換ツール（5.6節）も用意する。ただしブラウザ変換ツールも「変換してquestions.jsonをダウンロードする」ところまでで、そのファイルを`site/src/data/questions.json`に配置してビルド・デプロイし直す工程（④⑤）は変わらず手動である（サーバーサイド処理・認証機構を持ち込まない静的サイトの前提は維持する）。

### 5.2 Excelフォーマット（実データに基づき確定）

サンプルファイル `20260627_科目3_社会文化地域_言語と社会_問題_v1_問題.xlsx` を解析した結果、既存の問題作成スキルの出力は以下の構成であることを確認した。**1ファイル＝1つの大分類・中分類（科目）**を表し、ファイル内で複数科目を混在させない運用になっている。

**ファイル名／シート「管理データ」の「タイトル」セルの命名規則**
```
{YYYYMMDD}_科目{N}_{大分類}_{中分類}_問題_v{バージョン}
例）20260627_科目3_社会文化地域_言語と社会_問題_v1
```
この文字列から `科目N`・大分類・中分類を抽出し、`category`/`subcategory` として使用する（漢字の区切り方に表記ゆれが出やすいため、5.4節の `category-master.json` で正式表記・IDスラッグに正規化する）。

**シート「問題」**（1行1問、変換の主対象）
| 列 | 項目 | 必須 | 変換での扱い |
|---|---|---|---|
| A | No | 必須 | ID生成・他シートとの突合キーに使用 |
| B | 難易度 | 必須 | `★`=1, `★★`=2, `★★★`=3 に変換して `difficulty` へ |
| C | 問題文 | 必須 | `text` |
| D〜G | 選択肢A〜D | 必須 | `choices[0..3]` |
| H | 回答 | ―（無視） | **正解データではない**。ユーザーが自己学習時にExcel上へ記入した解答記録であり、実データでも181問中35問しか埋まっておらずそのうち15問は不正解（誤答）だった。変換対象から除外する |
| I | 結果 | ―（無視） | 同上、自己採点メモ用の未使用列 |
| J | 解説（選択後に表示） | 任意 | `explanation`。正解・誤答の解説が結合された1セルなのでそのまま転記 |
| K | 出典 | 任意 | `source`（例：`教程1`〜`教程7`） |

**シート「管理データ」**（正解の正式ソース）
- 7行目以降に `No / 正解` の2列表（例：`1, C` `2, C` …）があり、これが**正解の正式データ**。シート「問題」の「回答」列とは突合しない。
- 1〜5行目にタイトル・総問題数・難易度別内訳のサマリーがあり、タイトル文字列からカテゴリを抽出する（上記参照）。

**シート「詳細解説」**（補助・整合性チェック用）
- `No / 難易度 / 問題文 / 正解 / 選択肢A〜D / なぜ正解か / 各誤答の解説 / 出典` の列を持つ。「問題」シートと重複する内容だが、「正解」列と「なぜ正解か」「各誤答の解説」が分離されている。
- サンプルでは本シートの「正解」列は「管理データ」シートと181問すべて一致していたため、**変換スクリプトは「管理データ」を正とし、「詳細解説」の正解列は整合性チェック（クロスバリデーション）にのみ使用**する（両者が食い違う行があればエラーとして報告）。

### 5.3 変換スクリプトのバリデーション要件
変換時に以下をチェックし、エラー一覧をコンソール出力（該当No付き）してから停止する：
- 「問題」シートの必須列（No/難易度/問題文/選択肢A〜D）の欠落
- 「管理データ」シートに全Noぶんの正解が揃っているか（欠番検知）
- 「管理データ」の正解と「詳細解説」の正解列が一致しているか（不一致があれば警告）
- 正解の値が `A`〜`D` の範囲外
- 選択肢が実質的に重複していないか（簡易チェック）
- ファイル名／タイトルからカテゴリ・中分類を抽出できない場合はエラー停止（`category-master.json` に未登録の科目コードの場合も警告）
- 生成される `id`（`{科目slug}-{中分類slug}-{No}`）が既存 `questions.json` の中で重複していないか
- 既存 `questions.json` に存在した `id` が今回の変換結果から消えていないか（削除検知。誤削除防止のため警告表示）

なお「問題」シートの「回答」「結果」列（自己学習の記入跡）は変換対象外だが、将来的に「Excel上で自己採点しながら解く」運用と「Webサイトでの演習」を併用したい場合は、この列を無視して構わないかを一度ユーザーに確認する（現状は無視する前提で設計）。

### 5.4 実装技術（変換スクリプト）
- Node.js + `exceljs`（または `xlsx`）でExcel読み込み
- `npm run convert -- path/to/questions.xlsx` のようなCLIコマンドとして実行し、複数ファイルを一括変換できるよう `questions-source/**/*.xlsx` のようなglob指定にも対応する
- カテゴリ名の正式表記・IDスラッグ変換テーブルとして `site/src/data/category-master.json` を用意し、ファイル名から抽出した「科目N・大分類・中分類」をこのマスタと突合して正規化する。このファイルはNode CLIとブラウザ版アップロードツール（5.6節）の両方から参照する単一の正であり、`scripts/` 配下には置かない

```jsonc
// site/src/data/category-master.json（例：中分類は科目ごとに複数登録できる）
{
  "科目3": {
    "category": "社会・文化・地域",
    "categorySlug": "s3",
    "subcategories": {
      "言語と社会": "gengo-shakai",
      "異文化理解": "ibunka-rikai"
    }
  },
  "科目4": {
    "category": "言語と教育",
    "categorySlug": "s4",
    "subcategories": {
      "教授法": "kyoju-ho"
    }
  }
}
```
`questions-source/科目3_社会文化地域/` 配下の中分類フォルダ名（例：`言語と社会`）は、この `subcategories` のキーと完全一致させる（5.5.1節）。
- 新しい科目コード・中分類が出てきた場合、変換スクリプトはエラー停止し「`category-master.json` に追記してください」とメッセージを出す（自動登録はしない。表記ゆれをスキルではなく人間が確定するため）
- 出力: `site/src/data/questions.json`（6.5節のフォルダ構成を参照）

### 5.5 ソースファイルの命名規則とフォルダ管理ルール

科目・中分類の数が増え、かつ同じ中分類を後から追記・修正してバージョンが上がっていくと「今どれが最新版か」「全体で何中分類ぶん揃っているか」が分からなくなる。これを防ぐため、`questions-source/` は下記のルールで固定運用する。

#### 5.5.1 フォルダ階層：科目 → 中分類 → current/archive

```
questions-source/
├─ index.md                              … 全カテゴリの状況一覧（人間用、5.5.4節・自動生成）
├─ index.json                            … 同内容の機械可読版（5.5.4節・自動生成）
│
├─ 科目1_言語/
│  └─ （中分類フォルダ）/
│     ├─ current/                        … 最新版1ファイルのみ（変換対象はここだけ）
│     └─ archive/                        … 旧バージョンの保管場所
│
├─ 科目3_社会文化地域/
│  └─ 言語と社会/
│     ├─ current/
│     │  └─ 科目3_社会文化地域_言語と社会_v01_20260627.xlsx
│     └─ archive/                        … 空（まだv1のみのため）
│
└─ 科目4_言語と教育/ … 科目5_言語一般/ 同様
```

ルール：
- `current/` には**1中分類につき常に1ファイルのみ**（最新版）を置く。変換スクリプトは `current/**/*.xlsx` だけをスキャン対象とし、`archive/` 配下は変換しない。
- 新しいバージョンができたら、旧ファイルを `current/` から `archive/` へ移動してから新ファイルを `current/` に置く（`/convert-questions` スキルが変換とあわせて自動でこの移動を行う。7.1節参照）。
- 中分類フォルダ名は `category-master.json` の `subcategories` に登録されている表記と完全一致させる。未登録の中分類フォルダを検出した場合、変換スクリプトはエラー停止する（表記ゆれの流入を防ぐ）。

#### 5.5.2 ファイル名規則

```
{科目N}_{大分類}_{中分類}_v{バージョン2桁}_{YYYYMMDD}.xlsx
例）科目3_社会文化地域_言語と社会_v01_20260627.xlsx
```

- 既存の問題作成スキルが出力するファイル名（`{YYYYMMDD}_科目{N}_..._v{n}_問題.xlsx`）とはフィールド順序・末尾が異なる。作成自体は今まで通りのファイル名で構わず、`/convert-questions` スキルが `current/` へ配置する際に上記の正規名へ自動リネームする（置き場所さえ合っていれば、細かいファイル名を人が意識しなくてよいようにする）。
- バージョン番号は**中分類単位の通し番号**（v01, v02, …）。内容を直したら必ずインクリメントする。
- **同じ中分類に問題を追加したいときは、新しいファイルを作らず現行の `current/` ファイルに行を追記する**。既存行の `No` は変更しない（`No` はID生成のキーであり、変えると過去の正誤履歴が「未回答」に戻ってしまうため。2.1節参照）。追記後はバージョン番号を1つ上げ、旧ファイルを `archive/` へ退避する。

#### 5.5.3 管理台帳の自動生成（全体状況をひと目で把握する）

`/convert-questions` 実行のたびに、以下2ファイルを自動更新する。Excelを1つずつ開かなくても、今どの中分類が何問あり最後にいつ変換したかを一覧できる。

```jsonc
// questions-source/index.json（例）
{
  "s3-gengo-shakai": {
    "category": "社会・文化・地域",
    "subcategory": "言語と社会",
    "currentFile": "科目3_社会文化地域/言語と社会/current/科目3_社会文化地域_言語と社会_v01_20260627.xlsx",
    "version": 1,
    "questionCount": 181,
    "lastConvertedAt": "2026-07-11T10:00:00+09:00"
  }
}
```
`index.md` はこれをMarkdownテーブルに整形したもの（科目／中分類／問題数／バージョン／最終変換日）。

#### 5.5.4 QAエージェントによる整合性チェック（7.3節と連動）
- `category-master.json` に登録済みなのに `current/` にファイルが存在しない中分類（着手漏れ）の検出
- `current/` に複数ファイルが存在する中分類（最新版が曖昧な状態）の検出
- フォルダ名・ファイル名・Excel内タイトルの3者が食い違っているケースの検出

### 5.6 ブラウザ版アップロード・変換ツール

Node CLI（`/convert-questions`、Claude Code経由）とは別に、**サイト上でExcelファイルを選択するとその場で `questions.json` に変換できる画面**を用意する。「データベースなし・静的サイト」という前提は変えず、変換処理はすべてブラウザ内（クライアントサイドJavaScript）で完結させる。サーバーへのアップロード・書き込みは行わない。

#### 5.6.1 画面と処理フロー
画面：`/import`（4章のサイトマップに記載。ナビゲーションから「問題を取り込む」等でアクセス。1.2節の通り私的利用サイトのため認証は設けない）

1. ユーザーがExcelファイル（5.2節と同じ問題／管理データ／詳細解説の3シート構成）を選択する
2. ブラウザ内で `xlsx`（SheetJS）ライブラリを使ってワークブックを読み込み、5.2節と同じ列マッピング・正解の突合ロジックをTypeScriptに移植したパーサーで解析する
3. `site/src/data/category-master.json`（6.5節。Node CLIとブラウザ側で共有する単一のマスタファイル）と突合し、大分類・中分類・IDスラッグを解決する
4. 5.3節と同じバリデーションを行い、エラー・警告を画面に一覧表示する
5. エラーがなければ、現在ブラウザにロードされている `questions.json`（ビルド時に同梱された最新版）とメモリ上でマージしたプレビュー（追加件数／更新件数）を表示する
6. 「ダウンロード」ボタンで、マージ後の `questions.json` をファイルとしてダウンロードする

#### 5.6.2 この機能でできないこと（設計上の制約）
- ダウンロードした `questions.json` を `site/src/data/questions.json` に置き換えてビルド・デプロイし直す工程は、引き続き手動（またはClaude Code経由）で行う。ブラウザから直接VPSやリポジトリへ書き込むことはできない（サーバーサイド処理を持たない静的サイトの前提を維持するため。9章）。
- `category-master.json` に未登録のカテゴリが検出された場合、ブラウザ側からその場でマスタファイルを更新することはできない（ブラウザにファイルシステムへの書き込み権限がないため）。エラーメッセージで「Claude Codeに伝えて `category-master.json` に追記してもらってください」等を案内するに留める。
- `questions-source/` フォルダへの原本Excelの配置・`archive/` への旧版退避・管理台帳（`index.md`/`index.json`、5.5.3節）の更新は、引き続きNode CLI（`/convert-questions`）側の責務とする。ブラウザ版はあくまで「その場で試しに変換してみる／questions.jsonを素早く更新する」ための軽量ツールという位置づけで、5.5節のフォルダ管理ルールを代替するものではない。

#### 5.6.3 実装技術
- `xlsx`（SheetJS）パッケージを `site/` に追加し、`<input type="file">` で選択したファイルを `ArrayBuffer` として読み込みパースする
- パーサー・バリデーションロジックは `scripts/lib/parseExcel.mjs`・`validate.mjs` と揃えるが、Node.js専用API（`node:fs` 等）に依存しないブラウザ向けTypeScript実装として `site/src/utils/excelImport.ts` に置く（Node版とロジックの二重管理にはなるが、Node CLIとブラウザで実行環境が根本的に異なるため許容する。将来的にロジック共通化の必要性が高まれば、共通パッケージへの切り出しを検討）
- カテゴリマスタは `site/src/data/category-master.json` を単一の正とし、Node CLI側（`scripts/convert-questions.mjs`）もこのファイルを相対パスで参照する形に変更する（6.5節）

## 6. インフラ・デプロイ構成

### 6.1 全体構成
- フロントエンド：静的サイト（ビルド後は HTML/CSS/JS の静的ファイル一式）
- ホスティング：さくらVPS上のnginx
- バックエンド：なし（API・DBサーバーなし）

### 6.2 技術スタック（推奨）
- ビルドツール：Vite
- フレームワーク：Vue 3（軽量・学習コスト低・単一ファイルコンポーネントで見通しが良いため推奨。React でも代替可）
- 状態管理：Piniaもしくは素の Composition API（規模的にPiniaは過剰な可能性もあり、必要に応じて判断）
- グラフ表示：Chart.js（カテゴリ別正答率などの可視化）
- スタイル：軽量CSSまたはUnoCSS/Tailwind（任意）

### 6.3 nginx設定要件
- `dist/` 配下の静的ファイルを配信
- SPAのため、存在しないパスは `index.html` にフォールバック（`try_files $uri $uri/ /index.html;`）
- gzip圧縮を有効化
- `questions.json` 更新時にブラウザキャッシュが古いまま残らないよう、ビルド時にファイル名へハッシュを付与する（Viteのデフォルト挙動を利用）か、`Cache-Control` を適切に設定
- HTTPS化（Let's Encrypt等、既存VPS運用に準拠）

### 6.4 デプロイ手順（運用ドキュメント化する項目）
1. `npm run convert` でExcel→JSON変換
2. `npm run build` で `dist/` 生成
3. `rsync -avz --delete dist/ user@vps:/path/to/site/` でVPSへ転送
4. （必要なら）nginxの `reload`

## 7. 開発体制：Claude Codeエージェント／スキル設計

このアプリの開発・運用を継続的にClaude Codeで行うことを前提に、以下の役割分担を用意する。

### 7.1 問題データ変換スキル（`/convert-questions`）
- 役割：指定されたExcelファイル（または `questions-source/` 全体）を受け取り、変換スクリプト（5.4節）を実行し、バリデーション結果を人間が読める形で報告する
- 挙動：
  - 新規または更新されたファイルを `questions-source/{科目}/{中分類}/current/` の正規ファイル名に配置し、既存の旧バージョンがあれば `archive/` へ移動する（5.5.1〜5.5.2節）
  - エラーがあれば該当No付きで一覧表示し、修正を促す（自動修正はしない。問題文の正誤に関わるため人間判断を必須とする）
  - 成功時は「追加された問題数」「更新された問題数」「削除が検知された問題ID（あれば警告）」を要約表示するとともに、`questions-source/index.md`/`index.json`（管理台帳）を再生成する
  - 引数なしで実行した場合は `current/` 配下の全ファイルを一括再変換し、`index.md` で全体状況を確認できるようにする
- 想定トリガー：「問題を変換して」「Excelを反映して」等のユーザー発話、または `.xlsx` ファイルを指しての依頼

### 7.2 デプロイ用サブエージェント（`deploy-agent`）※任意・スコープ外機能
VPSのセットアップ・公開作業はユーザーが別途自力で行う（Linuxサーバー、8章参照）ため、本プロジェクトとしてはビルド成果物（`site/dist/`）とnginx設定サンプル・デプロイ手順書（`deploy/`一式、6.5節）を用意するところまでを行い、実際のVPS接続・デプロイ実行は行わない。将来ユーザー自身が「Claude Codeにデプロイまで任せたい」と判断した場合に備えて、設計だけ以下に残しておく。
- 役割：ビルド〜VPSへの転送までを一括実行する
- 安全設計：
  - VPSへの転送（rsync）は外部システムへの反映を伴う不可逆性の高い操作のため、**実行前に必ず差分（変更されるファイル一覧）を提示し、ユーザーに確認を取ってから実行する**
  - 転送後、簡易ヘルスチェック（トップページのHTTPステータス確認）を行い、結果を報告する
- ツール権限：Bash（rsync/npm実行）に限定し、他ツールは不要

### 7.3 QA／データ整合性チェックエージェント（`qa-agent`）
- 役割：`questions.json` のスキーマ検証、選択肢数・正解インデックスの妥当性、カテゴリ名の表記ゆれ（例：「言語と社会」「言語・社会」のような揺れ）を検出
- 加えて `questions-source/` のフォルダ管理ルール違反も検出する（5.5.4節）：`current/` に複数ファイルがある中分類、`category-master.json` 未登録なのに存在する中分類フォルダ、登録済みなのに `current/` が空の中分類（着手漏れ）
- 実行タイミング：変換スキル実行後、またはデプロイ前の最終チェックとして自動的に走らせる

### 7.4 開発フェーズと各フェーズでの担当
| フェーズ | 内容 | 主な作業者 |
|---|---|---|
| Phase 1 | プロジェクト雛形構築（Vite+Vue雛形、ディレクトリ構成、ルーティング） | Claude Code（通常セッション） |
| Phase 2 | 出題機能（4択・即時採点・解説表示）実装 | 同上 |
| Phase 3 | 成績記録（localStorage設計の実装）・苦手克服モード実装 | 同上 |
| Phase 4 | ダッシュボード（正答率グラフ等）実装 | 同上 |
| Phase 5 | Excel変換スクリプト＋`/convert-questions`スキル実装 | 同上 |
| Phase 6 | エクスポート／インポート／リセット機能実装 | 同上 |
| Phase 7 | nginx設定サンプル・デプロイ手順書の整備（`deploy/`一式） | 同上（実際のVPSセットアップ・デプロイ実行はユーザーが別途自力で実施するためスコープ外） |
| Phase 8 | サンプル問題（数問〜数十問）を用いた通しテスト・実機（スマホ含む）確認 | 同上 + ユーザーによる実機確認 |
| Phase 9 | ブラウザ版アップロード・変換ツール実装（`/import`画面、5.6節）。あわせて `category-master.json` を `site/src/data/` へ移設しNode CLI側の参照先も更新 | 同上 |

各フェーズ完了時は `/verify` 相当の確認（実際にブラウザで機能を動かして確認）を行ってから次フェーズへ進む。

- [ ] ブラウザ版アップロード画面（`/import`）をトップナビに常時表示するか、目立たない場所に置くか（想定利用者は本人のみのため常時表示で問題ないと考えるが確認）

## 9. スコープ外（初期リリースでは対応しない）

- 出題数のユーザー自由指定（初期リリースは50問固定。3.5節）
- 課金・有料機能
